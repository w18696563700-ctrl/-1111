# 竞标服务费预授权 Gate 后移与项目沟通开放条件 Addendum V1

状态：Frozen

冻结日期：2026-05-07

## 0. 总裁决

普通展览项目竞标链路中，`4000 元竞标服务费预授权额度` gate 从“提交竞标前”后移到“发布方资料确认通过后、项目级自由发送开启前”。

本 Addendum 冻结以下主裁决：

1. 竞标方在参与申请审核通过后，可以先提交报价和三份竞标资料。
2. `bid submit` 不再受 `4000 元竞标服务费预授权额度` gate 阻断。
3. Server 必须先保存 Bid、报价字段和三份 confirmed 附件，再让发布方进入资料确认。
4. 发布方资料确认通过后，竞标方才收到完成 `4000 元竞标服务费预授权额度` 的提醒。
5. 竞标方完成 `4000 元竞标服务费预授权额度` 后，项目级自由发送开启。
6. 项目沟通入口可见不等于项目沟通可发送。
7. 资料确认不是 BidAward、成交、最终合同金额确认或扣款。
8. 消息楼只承接入口、提醒、沟通和状态展示，不持有交易真值。

## 1. Supersede 范围

本 Addendum 只 supersede 旧冻结单中以下片段：

`审核通过 -> 4000 元竞标服务费预授权额度 -> 提交竞标 -> 项目沟通`

supersede 后的新片段为：

`审核通过 -> 提交报价 + 三份竞标资料 -> 发布方资料确认 -> 资料确认通过 -> 4000 元竞标服务费预授权额度 -> 项目级自由发送开启`

本 Addendum 不推翻、不重开、不修改以下冻结内容：

1. 项目创建页、草稿编辑页、预发布补资料页、发布确认页。
2. `draft -> submitted -> published` 状态链。
3. 正式发布前的 `200 元项目真实性诚意金`。
4. 上传三步流：init -> direct upload -> confirm。
5. `FileAsset` / `Evidence` / `ProjectAttachment` 真值归属。
6. BidAward 主路径。
7. Order / Contract seed 语义。
8. 最终合同金额确认主路径。

## 2. 新主链路

普通展览项目竞标到项目级自由发送开启的最小链路冻结为：

1. 竞标方查看 published 项目。
2. 竞标方提交参与申请。
3. 发布方审核通过参与申请。
4. 竞标方上传并确认三份竞标资料。
5. 竞标方填写报价金额、报价说明和必要报价字段。
6. 竞标方提交竞标。
7. Server 保存 Bid、报价字段和三份资料附件真值。
8. 发布方在消息楼 / 项目沟通 workbench 收到资料确认待办。
9. 发布方打开资料确认单。
10. 发布方确认资料通过或要求补充。
11. 资料确认通过后，竞标方收到完成 `4000 元竞标服务费预授权额度` 的提醒。
12. 竞标方完成 `4000 元竞标服务费预授权额度`。
13. Server 判定 project communication thread 满足发送条件。
14. 项目级自由发送开启。

后续选择合作方、BidAward、Order / Contract seed、最终合同金额确认不属于本 Addendum 的施工范围。

## 3. Bid Submit 裁决

`POST /api/app/bid/submit` 不再以前置 `4000 元竞标服务费预授权额度` frozen 状态作为通过条件。

`bid submit` 仍必须满足 Server 校验，包括但不限于：

1. 当前参与方具备通过审核的参与资格。
2. 当前项目处于可竞标状态。
3. 三份竞标资料附件均已完成 FileAsset confirm，且归属、用途和项目上下文合法。
4. 报价金额、报价说明、报价有效期等字段合法。
5. 未重复提交；重复请求必须受幂等或重复提交保护。
6. 当前账号、组织、角色和项目权限合法。
7. Server 现有风险、审计、状态机和资源可用性校验通过。

`bid submit` 成功只表示竞标方完成报价与资料提交，并由 Server 保存竞标提交真值。

`bid submit` 成功不表示：

1. 已完成 `4000 元竞标服务费预授权额度`。
2. 项目级自由发送已开启。
3. 发布方已确认资料。
4. BidAward 已成立。
5. 已成交。
6. 最终合同金额已确认。
7. 已扣款或已产生服务费结算。

## 4. 资料确认裁决

发布方资料确认是 `bid submit` 之后、`4000 元竞标服务费预授权额度` 之前的业务节点。

资料确认通过只代表：

1. 发布方认可竞标方提交的报价依据资料可进入后续沟通。
2. 平台可以向竞标方发出完成 `4000 元竞标服务费预授权额度` 的提醒。
3. 项目沟通可以进入“待预授权后开启自由发送”的状态。

资料确认通过不代表：

1. BidAward。
2. 成交。
3. 合同金额确认。
4. 最终合同金额确认。
5. Order / Contract seed 已生成。
6. 服务费扣款。
7. 保证金冻结。
8. 支付、结算、发票或钱包状态变化。

资料要求补充时，项目级自由发送不得开启；竞标方应先补充资料并重新进入资料确认。

## 5. 项目沟通假开真锁裁决

本 Addendum 将“项目沟通正式开启”精确定义为：

`项目级自由发送开启`

项目沟通入口可见不等于项目沟通可发送。

在项目级自由发送开启前：

1. 消息楼项目沟通入口可以展示。
2. 聊天窗口可以展示项目、资料、待办、状态提示和软链接。
3. Flutter 可以允许用户输入本地草稿。
4. 用户点击发送时，如果 Server 判定未满足发送条件，不得真实发送。
5. 不得落库 project communication message。
6. 不得生成对方 unread。
7. 不得生成对方 notification。
8. 不得将本地草稿展示为已发送消息。
9. 必须给出明确系统提示。

项目级自由发送开启必须同时满足：

1. 参与申请已通过。
2. Bid 与三份竞标资料已由 Server 保存。
3. 发布方资料确认通过。
4. 竞标方完成 `4000 元竞标服务费预授权额度`。
5. Server 判定该 project communication thread `canSendMessage=true`。

Flutter 和 BFF 不得绕过 Server 的 `canSendMessage` 真值。

## 6. 服务费预授权新触发点

`4000 元竞标服务费预授权额度` 的新触发点为：

`发布方资料确认通过 -> 竞标方收到预授权提醒 -> 竞标方完成 4000 元竞标服务费预授权额度 -> 项目级自由发送开启`

预授权是额度确认，不是扣款。

预授权不得被描述为：

1. 已扣款。
2. 保证金。
3. 竞标报名费。
4. 成交服务费已收取。
5. 最终合同金额的一部分。

未中标释放、发布方毁约释放、项目条件重大变化释放、服务费扣取、结算、退款等规则不在本 Addendum 展开，必须另行冻结。

## 7. Server / BFF / Flutter 边界

### 7.1 Server

Server 是以下真值 owner：

1. 参与申请状态。
2. Bid 提交状态。
3. 三份竞标资料附件归属与 confirmed 状态。
4. 发布方资料确认结果。
5. `4000 元竞标服务费预授权额度` 状态。
6. project communication thread 是否允许发送。
7. 通知、待办、unread 和审计留痕。

Server 必须在写项目沟通消息前校验发送条件。

### 7.2 BFF

BFF 只做：

1. App-facing 聚合。
2. 鉴权上下文透传。
3. Server read model 透出。
4. routeTarget、chatAvailability、通知和待办 projection。
5. 错误归一化。

BFF 不得：

1. 持有预授权真值。
2. 自行放行项目沟通发送。
3. 创造资料确认状态。
4. 创造交易状态。
5. 持有第二套状态机。

### 7.3 Flutter

Flutter 只做：

1. 提交报价与三份资料的表单展示和提交。
2. 资料确认状态展示。
3. 预授权提醒展示和跳转。
4. 项目沟通入口展示。
5. 本地草稿输入。
6. 根据 Server / BFF projection 展示锁定提示。
7. 发送失败或未开放时展示明确提示。

Flutter 不得：

1. 本地模拟预授权完成。
2. 本地模拟项目级自由发送已开启。
3. 本地模拟已发送消息。
4. 绕过 Server truth。
5. 把聊天内容当成交易真值。

## 8. 测试环境放行原则

本 Addendum 只冻结测试环境 Server-owned test authorization 的原则，不冻结具体接口、表结构、字段、后台工具或实现方式。

测试环境放行必须满足：

1. 只能由 Server 持有和判定。
2. 只能用于测试账号。
3. 只能用于测试项目。
4. 必须具备 TTL。
5. 必须具备审计记录。
6. 必须具备环境隔离。
7. 不得允许生产用户触发。
8. 不得由 Flutter 本地绕过。
9. 不得由 BFF 自行放行。

具体接口、表、配置、Admin 工具或沙箱通道实现方式，必须后续单独裁决。

## 9. 失效提醒治理边界

消息铃铛失效入口治理不纳入本 Addendum 主施工范围。

以下能力后续单独开“消息通知治理专项”裁决：

1. 失效 routeTarget 自动检测。
2. 失效提醒批量 mark read / dismiss。
3. 失效提醒是否从 unread 排除。
4. 失效提醒是否默认隐藏或折叠。
5. 失效提醒历史审计保留规则。
6. 通知 dismissed / expired / invalid_target 语义。

本 Addendum 只要求：项目沟通 gate 后移和自由发送开启，不得借机扩大消息中心治理范围。

## 10. No-Go 范围

本 Addendum 不允许扩展以下能力：

1. 支付扣款。
2. 服务费结算。
3. 保证金。
4. 钱包。
5. 发票。
6. 退款。
7. 最终合同金额确认。
8. BidAward 主路径变更。
9. Order / Contract seed 语义变更。
10. 履约。
11. 验收。
12. 评价。
13. 争议。
14. IM / 私信 / 群聊 / 客服系统。
15. Admin 交易定标台。

## 11. 后续实施门禁

进入代码施工前，必须完成以下门禁裁决：

1. OpenAPI 是否同步移除 bid submit 前置预授权 409 语义。
2. generated types 是否需要更新。
3. project communication `chatAvailability` 是否新增待预授权 lock reason / next action。
4. Server 如何表达资料确认通过后的预授权提醒。
5. Server 如何判定项目级自由发送开启。
6. 测试环境 Server-owned test authorization 是否需要单独 SSOT / contracts / DB 裁决。

未完成上述门禁前，不得修改 Server、BFF、Flutter、OpenAPI、generated 或 DB。
