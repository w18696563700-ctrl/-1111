# Project Attachment Pending Kind Ownership And Message Confirmation Same Source Addendum

状态：本轮冻结

## 1. 本轮最小闭环

本轮只修正报价依据资料上传前的 Flutter 临时草稿态归属，并冻结消息楼资料确认的同源边界。

最小闭环：

- 预发布补资料页中，待上传附件必须记录所属资料类型。
- 切换资料类型只切换当前视图，不得移动已有待上传附件。
- 上传绑定时必须使用附件自己的资料类型，不得使用当前选中态覆盖。
- 上传成功后仍以后端正式附件列表回读为准。
- 消息楼资料确认只读取正式资料，不读取 Flutter 临时待上传附件。

## 2. 资料状态分层

| 状态层 | 所属层 | 真源 | 本轮规则 |
| --- | --- | --- | --- |
| 待上传附件 | Flutter 临时草稿态 | 用户本机刚选择的文件与当前资料类型 | 只用于上传前展示和操作，不是业务真相 |
| 正式报价依据资料 | Server 业务真相 | FileAsset / Evidence / ProjectAttachment | 上传三步流完成并绑定后，由后端读侧回读 |
| 消息楼资料确认 | 消息楼展示与确认承接 | 正式报价依据资料与确认状态 | 不读取 Flutter 临时草稿，不生成第二套资料真相 |

## 3. 字段与归属规则

| 项 | 规则 |
| --- | --- |
| `attachmentKind` | 用户选择文件时记录当时的资料类型 |
| `pending drafts` | 按资料类型分组保存 |
| checklist 展示 | 每个资料项只显示自己名下的待上传附件 |
| 切换 chip | 只改变当前选中资料类型，不移动待上传附件 |
| 更换资料类型 | 如需移动附件归属，必须走显式操作，不得因切换 chip 自动移动 |
| 上传绑定 | 每个附件使用自己的 `attachmentKind` 绑定 |
| 正式列表 | 以后端正式附件列表回读为准 |

五类资料类型保持不变：

- 效果图。
- 尺寸图 / 施工图。
- 材质图 / 材料样板。
- 设备物料清单。
- 服务清单。

## 4. 消息楼同源边界

消息楼中的发布方资料、五类资料确认、下一页确认入口，必须基于正式资料状态推进。

冻结规则：

- 消息楼不读取预发布页 Flutter 临时 pending drafts。
- 消息楼不复制、不伪造、不另建文件真相。
- 消息楼资料确认应基于正式附件 ID、资料类型、确认状态推进。
- 若消息楼当前真实正式附件链路未完全打通，本轮只能标记为后续联调项，不得伪装为已联通。

## 5. 按钮能力冻结

以下能力必须保留：

- 选择附件。
- 继续添加附件。
- 预览。
- 打开。
- 移除。
- 更换资料类型。
- 上传并形成正式附件。
- 刷新正式列表。

上传三步流保持不变：

1. init
2. direct upload
3. confirm / bind

## 6. 不做事项

本轮不做以下事项：

- 不改 BFF。
- 不改 Server。
- 不改 OpenAPI / contracts。
- 不改数据库。
- 不改上传三步流。
- 不改 FileAsset / Evidence / ProjectAttachment 真值。
- 不改消息楼确认状态机。
- 不新增资料类型。
- 不把一份临时文件自动挂到多个资料类型。
- 不让消息楼读取 Flutter 临时草稿态。

## 7. 验收标准

- 上传一个文件后，只出现在选择时对应的资料类型下面。
- 切换到其他资料类型，不再显示同一份待上传文件。
- 如果不同资料类型下各自选择附件，上传时逐个使用各自资料类型绑定。
- 移除附件只影响该附件所属资料类型。
- 上传后正式资料仍以后端回读为准。
- 消息楼不产生第二套资料真相。
- `flutter analyze --no-pub` 无本轮新增问题。
- `project_attachment_corridor_test.dart` 通过，或明确记录既有环境阻塞。

## 8. 风险与降级

- 如果消息楼正式资料链路当前不可读，本轮只标记为后续联调项。
- 如果本地 Flutter 环境再次出现 native asset hook 缓存问题，只允许清理本地生成缓存，不得删除源码。
- 如果附件组件被其他页面复用，修复必须控制在当前上传前 pending draft 归属，不得改动竞标提交或消息楼确认状态机。
