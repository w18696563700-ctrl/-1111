# 发布-竞标-沟通-合作确认冲突清单 V1

## 1. 冻结结论

当前存在历史文书、旧 UI 文案和当前平台定价主线之间的口径冲突。本文件只做冲突登记和本轮解释优先级冻结，不解锁 BFF、Server、contracts、OpenAPI、数据库修改。

## 2. 冲突清单

| 编号 | 冲突点 | 当前风险 | 本轮处理 |
| --- | --- | --- | --- |
| C-01 | 旧 `P0-Pay / 发单诚意金 / 询价报价单 / 3%` 与当前 `200 / 4000 / 成交确认` 主线并存 | 用户误以为当前普通发布链路仍按旧交易任务收费 | 普通发布链路主体验只显示当前口径；旧口径保留为历史兼容文书和内部实现痕迹 |
| C-02 | `200 元` 曾被写作发单诚意金、冻结、押金等 | 资金性质误导，影响信任 | 统一为 `项目真实性诚意金`，不得写押金、罚款、平台服务费 |
| C-03 | `选择合作方` 文案过轻，但后端实质是 `BidAward -> Order -> Contract seed` | 发布方误以为只是收藏或标记候选方 | 用户侧改为 `发起合作确认 / 确认合作并生成订单`，并提示后续进入订单/合同承接 |
| C-04 | BidAward 文书存在 `awarded -> converted_to_order` 中间态，当前合同生成类型主要暴露 `converted_to_order` | 前端若展示中间态会造成状态漂移 | 本轮不展示稳定中间态，按返回的订单/合同锚点承接 |
| C-05 | 消息楼有项目沟通、竞标线程、软链接，但不是业务真相 owner | 容易被误扩成支付台、订单台、裁决台 | 消息楼只做沟通、结果通知、软链接，不提交定标或成交命令 |
| C-06 | `converted_to_order` 后报价依据资料读取可能受项目状态限制 | 合作后资料可见性可能被 Server/BFF 规则阻断 | 本轮登记为后续 Server/BFF 解锁建议，不在 Flutter 伪造资料可见状态 |
| C-07 | 项目详情、竞标提交、弹窗存在 `BFF/Server/Flutter/handoff/routeTarget` 技术文案 | 普通用户理解成本高，像工程验收页 | 本轮前端只改可见文案，不改路由、接口、状态 |
| C-08 | 竞标方 `4000 元` 预授权容易被误读为已扣费 | 商业闭环理解偏差 | 展示为 `竞标服务费预授权额度`，说明成交后按规则处理，未成交按规则释放 |

## 3. 本轮废弃 / 降级口径

以下口径不得作为当前普通发布链路主体验文案：

1. `P0-Pay 交易任务`
2. `发单诚意金`
3. `询价报价单`
4. `明价竞标单`
5. 固定 `3%` 作为用户主体验收费说明
6. `Flutter 只提交...`
7. `BFF/Server 聚合...`
8. `routeTarget / handoff / FileAsset ID / objectKey`

说明：上述词不要求在历史文书、底层类名、兼容代码中立刻消失；本轮只要求普通用户主体验不再以它们作为解释主线。

## 4. 不做事项

1. 不修改 BidAward 后端实现。
2. 不修改 `select-bid-and-create-order` 或 `/api/app/bid/award` 路由。
3. 不修改消息楼 routeTarget 注册表。
4. 不新增资料可见性字段。
5. 不新增待办、统计、评价、比较、合同确认假入口。
6. 不把 mock 当生产数据。

## 5. 进入实现条件

只有同时满足以下条件，才允许进入 Flutter 展示层改造：

1. 本轮文书已写清最小闭环和禁止事项。
2. 实现范围限定在 Flutter 展示层和用户文案。
3. BFF、Server、contracts、OpenAPI、数据库保持只读。
4. 任何发现需要后端真值变化的问题，转为下一轮解锁建议，不在本轮硬改。
