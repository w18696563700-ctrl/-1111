# 支付 / 扣费 / 保证金 / 结算 / 发票 / 钱包能力蓝图 V1

## 0. 总裁决

资金能力必须分模块、分阶段进入冻结和施工。消息楼只是资金动作的提醒和跳转枢纽，真实资金动作必须在我的楼、我的项目、我的竞标、合同/订单/账单等真实页面完成，并由 Server 持有业务真值。

本文件是 P1/P2 蓝图，不解锁真实支付宝/微信预授权完整接入、自动解冻定时任务、扣费、保证金、结算、发票或钱包施工。

## 1. 当前最小闭环

| 能力 | 当前 P0 裁决 |
|---|---|
| 4000 竞标服务费预授权 | 后移到资料确认通过后、项目级自由发送开启前；当前只完成门禁位置和提醒 |
| 预授权测试放行 | 只冻结 Server-owned 原则；具体接口/表/实现另开裁决 |
| 项目详情资金说明 | 只读折叠展示，不执行资金动作 |
| 消息楼 | 提醒、入口、跳转；不执行支付、不扣费 |
| 我的楼 | 真实资金状态和后续业务页的独立入口承接方 |

## 2. 分模块蓝图

| 模块 | 真值 owner | 真实页面 | P0/P1/P2 | 说明 |
|---|---|---|---|---|
| 支付通道预授权 | Server | 我的竞标 / 预授权记录页 | P1 | 支付宝/微信通道、回调、幂等、状态读回 |
| 自动解冻 | Server + job | 我的竞标 / 资金状态页 | P1 | 定时规则、过期释放、审计 |
| 服务费扣费 | Server | 合同 / 账单 / 支付状态页 | P1 | 读取 finalConfirmedAmount 后计算和扣费 |
| 履约保证金 | Server | 我的信用与保证金 | P2 | 独立保证金状态机，不混预授权 |
| 结算 | Server | 账单 / 结算页 | P2 | 清分、对账、结算批次 |
| 发票 | Server | 发票页 | P2 | 发票申请、开具、状态、税务信息 |
| 钱包 | Server | 钱包页 | P2/No-Go 当前 | 不在 P0/P1 早期引入资金池 |

## 3. 边界

1. 预授权不是扣款。
2. 服务费扣费不得早于最终合同金额确认。
3. 保证金不得复用竞标服务费预授权状态机。
4. 钱包不得成为隐式资金池。
5. Flutter 不得绕过 Server truth。
6. BFF 不得持有资金状态机。
7. 消息楼不得成为支付执行台。

## 4. 最稳路径

1. 先完成当前 P0：资料确认 -> 预授权提醒 -> 未预授权锁发送 -> test authorization blocker 明确。
2. 再单独冻结 test authorization / 支付沙箱。
3. 再接真实支付宝/微信预授权通道。
4. 再做自动解冻。
5. 再接最终合同金额后的服务费扣费。
6. 最后分期进入保证金、结算、发票、钱包。

## 5. No-Go

1. 不一次性建设大支付中台。
2. 不在消息楼内直接支付、扣费、开票或钱包充值。
3. 不把预授权完成写成已付款。
4. 不把资料确认写成成交。
5. 不在没有 finalConfirmedAmount 的情况下计算最终服务费。

## 6. 当前阶段结论

当前最适合继续推进的是 P0 gate 后移和项目级自由发送锁定闭环。完整支付 / 扣费 / 保证金 / 结算 / 发票 / 钱包必须拆成独立专项，逐个走 SSOT、contracts、generated、Server、BFF、Flutter、云端 UAT 门禁。
