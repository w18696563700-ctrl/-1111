# 项目创建到发布主流程定义 addendum

状态：本轮冻结

## 1. 本轮最小闭环

本轮只理顺“创建项目 -> 草稿 -> 预发布信息补充 -> 补齐报价依据资料 -> 诚意金绿色通道表态 -> 确认发布 -> 竞标中”的页面定义、展示文案和 Flutter 展示层门禁。

本轮不删除现有页面能力，不做大重构，不改变真实接口、状态机、数据库或云端运行方式。

运行结构按当前项目总控口径执行：

- 本地只运行 Flutter 前端。
- BFF / Server 在阿里云。
- 本地联调统一通过 SSH 隧道访问 `http://127.0.0.1:8080`。
- 不把本地 3000 / 3001 当作真实 BFF / Server。

## 2. 当前主流程

| 顺序 | 页面 / 节点 | 当前状态 | 页面职责 | 主要动作 |
| --- | --- | --- | --- | --- |
| 1 | 创建项目 | 创建前 | 填写基础信息并保存为草稿 | 创建项目 |
| 2 | 我的项目 - 草稿列表 | `draft` | 承接未正式提交的项目 | 查看详情、继续编辑 |
| 3 | 编辑项目 草稿 -> 预发布信息补充页 | `draft` | 可编辑基础信息、地点、计划时间、补充说明，并确认进入预发布 | 仅保存草稿、确认保存并进入预发布信息补充页 |
| 4 | 我的项目 - 预发布列表 | `submitted` | 承接已进入发布前核对的项目 | 补资料后确认发布 |
| 5 | 预发布补资料并发布页 | `submitted` | 补充报价依据资料、完成诚意金绿色通道表态、确认发布 | 上传资料、支持 / 暂不支持表态、确认并发布 |
| 6 | 竞标中 | `published` | 项目进入公域竞标展示 | 查看详情、按真实能力补充资料或撤回 |

当前主状态链路冻结为：

`draft -> submitted -> published`

## 3. 页面命名冻结

| 旧心智 / 易混点 | 本轮统一展示 |
| --- | --- |
| 草稿编辑进入“预发布列表” | `编辑项目 草稿 -> 预发布信息补充页` |
| 预发布列表详情 / 我的项目详情 | `预发布补资料并发布页` |
| 预发布确认动作 | `确认保存并进入预发布信息补充页` |
| 正式发布动作 | `确认并发布` |

命名只作为 Flutter 展示文案，不改变 Server 状态名、接口名或真实 action。

## 4. 资料门禁规则

本轮前端展示层发布门禁调整为三类必传资料：

| 资料类型 | 字段 / kind | 门禁 | 说明 |
| --- | --- | --- | --- |
| 效果图 | `effect_image` | 必传 | 发布前必须至少形成 1 个正式附件 |
| 尺寸图 / 施工图 | `construction_doc` | 必传 | 发布前必须至少形成 1 个正式附件 |
| 材质图 / 材料样板 | `material_sample` | 必传 | 发布前必须至少形成 1 个正式附件 |
| 设备物料清单 | `equipment_material_list` | 建议补充 | 保留入口和能力，不作为本轮硬门槛 |
| 服务清单 | `service_list` | 建议补充 | 保留入口和能力，不作为本轮硬门槛 |

资料门禁只使用正式附件列表回读，不使用待上传草稿附件作为已满足条件。上传三步流仍为 `init -> direct upload -> confirm / bind`，`FileAsset / Evidence` 真值不变。

如果后续需要 Server 也强制三类资料门禁，必须另开 contracts / BFF / Server 冻结轮次。本轮不得声称后端已经完成三类资料强拦截。

## 5. 诚意金绿色通道规则

项目真实性诚意金流程必须存在，但 App 上线初期不在 Flutter 展示层硬性要求真实支付或冻结。

本轮冻结为：

- 诚意金模块在“报价依据资料”之后展示。
- 必须提供二选一表态：
  - `支持项目真实性诚意金机制`
  - `暂不支持项目真实性诚意金机制`
- 未选择时，底部 CTA 不允许进入确认发布。
- 选择任意一项后，Flutter 展示层允许继续确认发布。
- 选择 `暂不支持项目真实性诚意金机制` 仍允许发布，当前仅作为内测民调 / 表态数据。
- 现有诚意金状态、刷新、继续处理支付 / 订单能力保留，不删除。

当前代码和云端可能仍存在 Server 端诚意金发布门禁。若 Server runtime 在发布接口仍返回 `PROJECT_AUTHENTICITY_SINCERITY_REQUIRED`，本轮只能如实展示失败，不能在 Flutter、BFF 或本地 mock 伪造发布成功。

## 6. 底部 CTA 规则

预发布补资料并发布页底部 CTA 按真实回读状态组合展示：

| 条件 | CTA |
| --- | --- |
| 三类必传资料未满足 | `补充报价依据资料` |
| 三类必传资料满足，但未完成诚意金绿色通道表态 | `完成诚意金绿色通道表态` |
| 三类必传资料满足，且已完成支持 / 暂不支持表态 | `确认并发布` |

确认发布弹窗：

- 标题：`确认发布项目`
- 主按钮：`确认发布`
- 次按钮：`返回继续核对`

## 7. 不做事项

本轮不做：

- 不改 BFF。
- 不改 Server。
- 不改 OpenAPI / contracts。
- 不改数据库 / migration。
- 不改项目状态机。
- 不改保存草稿、提交预发布、正式发布接口。
- 不新增 fake 状态、fake 支付、fake 附件、fake 发布能力。
- 不删除现有上传、预览、打开、移除、刷新、继续处理诚意金等功能。
- 不把本地 mock 或本地服务当云端真实能力。

## 8. 后续扩展位

需要另开轮次冻结的扩展位：

- Server 真正强制三类资料门禁。
- 诚意金绿色通道表态独立字段 / 独立状态。
- 诚意金上线正式期是否恢复真实支付 / 冻结。
- 发布确认后竞标中公域展示的资料可见性进一步细分。
- Admin 对资料完整度、诚意金表态、项目真实性风险的治理视图。

## 9. Gate 结论

本轮允许进入 Flutter 展示层轻量收敛，前提是：

- 只改 Flutter 展示和本 addendum。
- 不改 BFF / Server / contracts / 数据库 / 云端。
- 不删除现有好用功能。
- 对 Server 仍可能存在诚意金硬门禁的风险必须在验收回执中单独说明。
