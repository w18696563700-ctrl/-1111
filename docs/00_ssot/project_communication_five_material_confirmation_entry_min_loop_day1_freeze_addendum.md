# 《项目沟通五类资料确认入口最小闭环冻结单》

## 1. 总裁决

本轮裁决为 `Conditional Pass`。

允许进入第 2 天 Flutter 页面结构施工图，但不得直接进入代码实现。

通过条件是：本轮只冻结项目沟通页内的五类资料确认固定入口，不新增 BFF / Server 字段，不新增云端状态机，不把本地 BFF / Server 当作 active runtime 真相。

## 2. 当前最小闭环

项目沟通页的 `项目工作入口` 继续保留为项目级工作区，不改造成通用聊天框。

第 2 天施工图允许在该区域新增 `资料确认` 固定区，展示 5 个固定按钮：

- `效果图确认`
- `材质图确认`
- `尺寸图确认`
- `设备物料清单确认`
- `服务清单确认`

按钮状态口径冻结为：

- `未提交`：当前资料类型在报价依据资料包中无可读资料。
- `待确认`：当前资料类型已有可读资料，但本轮 Flutter 最小闭环尚无正式独立确认状态可证明已确认。
- `已确认`：仅在现有合同或既有消息/软链接读模型能明确给出已确认语义时展示；不得由 Flutter 靠文件存在、中文标题或本地猜测伪造。

视觉口径冻结为：

- `已确认` 使用绿色。
- `待确认` 使用主色或提醒色。
- `未提交` 使用中性禁用或弱化样式。

## 3. 需要保留但暂不开通

以下能力保留为扩展位，本轮不打开：

- 五类资料独立确认持久化。
- 每类资料确认人、确认时间、组织身份审计。
- 资料更新后自动重新进入待确认。
- 撤回确认、重新确认、多版本比对。
- Admin 侧争议核验入口。
- BFF / Server 专门的五类资料确认状态聚合字段。

## 4. 后续扩展位

后续若要把五类资料确认做成完整平台真值，必须另起合同和后端阶段，至少冻结：

- `attachmentKind` 对应确认对象的唯一锚点。
- `confirmationState` 枚举。
- `confirmedByOrganizationId` / `confirmedByUserId`。
- `confirmedAt`。
- 资料变更后的确认失效规则。
- 审计与争议读取边界。

该扩展必须由 Server 拥有业务真值，BFF 只做 App-facing 聚合，Flutter 只做消费和展示。

## 5. 与现有真相的关系

现有 `报价依据资料包 V1` 已冻结五类资料枚举：

- `effect_image`
- `construction_doc`
- `material_sample`
- `equipment_material_list`
- `service_list`

本冻结单只规定这些资料在项目沟通页中的固定确认入口展示名，不替换 `attachmentKind` 合同枚举。

现有 `项目沟通工作台 V1` 曾把确认卡定义为消息流中的工作确认卡，并限制为：

- `quote`
- `material_process`
- `schedule`

本冻结单对其作出阶段性收敛：五类资料确认不作为新的聊天输入主操作，不从底部输入栏发起，不新增五种 `confirmation_card` 类型。旧确认卡能力若仍存在，只能作为兼容读回或历史记录，不得成为五类资料确认的主入口。

## 6. 第 2 天准入边界

允许第 2 天做 Flutter 页面结构施工图：

- 保留顶部当前项目沟通卡。
- 保留 `进入审核`、`后续承接状态`、`项目相册` 三个既有入口。
- 在 `项目工作入口` 内新增 `资料确认` 固定区。
- 聊天记录区继续只展示沟通记录、附件消息和历史消息。
- 底部输入栏继续只承接文字、附件、图片等沟通输入。

不得在第 2 天做：

- Flutter 代码实现。
- BFF / Server 修改。
- contracts 修改。
- 云端写操作。
- 数据库结构、迁移、部署、服务重启。

## 7. Contract 结论

本轮不要求新增 contract 字段。

原因：

- 五类资料枚举和竞标方只读资料投影已存在。
- 现有合同未提供五类资料独立确认状态。
- 若第 2 天施工图需要展示 `已确认` 的真实持久化状态，必须先回到 Gate 1 新增合同冻结；不得由 Flutter 临场扩展字段或猜状态。

第 2 天可接受的低风险状态来源：

- `未提交`：来自五类资料只读列表中缺少对应 `attachmentKind`。
- `待确认`：来自五类资料只读列表中存在对应 `attachmentKind`，但无正式独立确认状态。
- `已确认`：仅来自已冻结合同字段或既有确认软链接读模型明确返回的已确认语义。

## 8. 分层边界

Flutter：

- 允许做固定入口 UI、状态展示、点击反馈和受控跳转。
- 禁止写业务真值、发明新 DTO、绕过 BFF 调 Server。

BFF：

- 本轮不改。
- 后续如扩展，只能做 App-facing 聚合和错误归一。

Server：

- 本轮不改。
- 后续如扩展，必须拥有五类资料确认真值、审计和失效规则。

Cloud runtime：

- 本轮不动云端。
- 后续只允许按授权做只读健康检查或明确批准的 smoke。

## 9. 四类判断

- 更稳：全链路五类资料确认状态由 Server 持久化并审计。
- 更省成本：本轮只冻结入口和 Flutter 施工图，不改 BFF / Server。
- 更适合当前阶段：先做 `资料确认` 固定入口的 Flutter 最小闭环，状态不足时降级为 `未提交 / 待确认`。
- 风险更大：继续把五类资料确认作为聊天输入确认卡发送，或未经合同冻结直接新增 BFF / Server 状态。

## 10. Day 1 Go / No-Go

Day 1 结论：

- 五类按钮命名已冻结。
- 确认入口不得作为聊天输入主操作。
- 本轮不新增后端状态机。
- 当前合同已有五类资料枚举，但没有五类资料独立确认状态字段。

第 2 天为 `Conditional Go`：

- 可以进入 Flutter 页面结构施工图。
- 不可以进入代码实现。
- 若施工图要求真实 `已确认` 持久状态，必须先新增 contracts / Server truth 冻结。
