# 《我的楼 V2.1 信用 / 保证金 / 交易保障 package 边界判断》

## A. Current Object

- 当前对象：
  - `我的楼专项开发主线`
  - `V2.1 信用 / 保证金 / 交易保障`
- 当前裁决类型：
  - package boundary judgment only

## B. Upstream Intake

- 当前必须承接以下现行前提：
  - `我的楼` 仍是 compact current-user hub
  - `我的项目` 仍是独立私域入口，不得被吞并
  - `V2.0 paid membership` 已完成 bounded implementation 成立判断
  - `payment.*` 当前仍是 pre-embed / reserve，不是现成 runtime payment system
  - 现有 `blacklist / whitelist / permanent-ban` 规则只构成治理基线素材，不自动等于 `V2.1` package 已冻结
- 当前还必须承接以下边界：
  - `V2.0 paid membership` 只解决商业权益 / 费率档位 / quota / upgrade guidance
  - `V2.0` 已明确排除：
    - guarantee
    - payment
    - billing
    - invoice
    - settlement
    - dispute / governance
  - `我的项目` 继续只承接当前组织 scope 下的私域项目资产与单项目私域推进真相
  - `profile` 继续只是 entry owner，不自动成为交易约束或交易保障 truth owner

## C. Current Judgment

- 当前 judgment：
  - `通过`
- 当前正式判断如下：
  - `V2.1 信用 / 保证金 / 交易保障` 构成《我的楼》当前合法下一包
  - 但它当前只允许进入：
    - `minimum package boundary freeze`
  - 当前不允许进入：
    - rules freeze
    - contracts freeze
    - backend truth freeze
    - BFF surface freeze
    - frontend surface freeze
    - implementation unlock
    - runtime implementation

## D. Current Minimum Meaning

- 若 `V2.1` 成立，它当前最小只允许被理解为：
  - 信用 / 保证金 / 交易保障 的 `rule-and-status boundary package`
  - 交易资格约束、履约约束、状态表达、规则 handoff、私域 status entry 的最小边界包
- 它当前不是：
  - runtime transaction package
  - 资金执行包
  - payment / billing package
  - admin governance console package
  - 项目交易主流程 package
- 当前最小语义切分写死如下：
  - `信用`
    - 只指交易约束、履约约束、限制状态、资格提示、规则说明的 boundary 语义
  - `保证金`
    - 只指保证金 requirement / eligibility / status / handoff 的 boundary 语义
  - `交易保障`
    - 只指保障资格、保障要求、保障状态、保障规则 handoff 的 boundary 语义
- 上述三者当前都不得被写成：
  - 资金冻结已执行
  - 扣罚已执行
  - 赔付已执行
  - 退款已执行
  - 清算已执行

## E. Included Scope At Boundary-Judgment Level

- 当前 judgment 层只允许纳入：
  - 交易约束与履约约束的 package 语义定义
  - `信用 / 保证金 / 交易保障` 的 boundary family 名称与对象边界
  - `我的楼` 下可能承接的 bounded private entry 方向
  - 与 `V2.0`、`V2.2`、项目、公域交易、admin governance 的切分边界
  - 明确 future dependency 的 retained hooks

### E.1 Boundary With V2.0

- `V2.0 paid membership` 继续只解决：
  - 商业权益
  - 费率档位
  - quota
  - upgrade guidance
  - membership summary
- `V2.1` 当前解决的不是会员权益，而是：
  - 交易资格约束
  - 履约约束
  - 保证金 requirement posture
  - 交易保障 eligibility / restriction posture
- 当前明确禁止混写：
  - 会员等级 = 交易资格
  - 会员付费状态 = 保证金已缴
  - 会员 entitlement = 交易保障已生效

### E.2 Boundary With V2.2

- `V2.1` 当前只允许先冻结：
  - 保证金规则
  - 保证金 requirement / status / handoff
  - 交易保障规则
  - 资格约束与约束状态
- `V2.2 payment / billing` 未来才承接：
  - 真实资金冻结
  - 扣罚
  - 赔付
  - 退款
  - 代收
  - 账单 / 发票
  - 清算 / 结算
- 因此当前 `V2.1` 只允许保留：
  - payment dependency hook
  - billing dependency hook
  - settlement dependency hook
  - not current package truth

### E.3 Boundary With Project / Public Trade / Admin Governance

- `V2.1` 当前不得吞并：
  - `我的项目` 私域主线
  - 公域交易浏览主线
  - 项目 / 订单 / 合同 / 履约 / 验收 canonical truths
- `V2.1` 当前也不得被写成：
  - 完整 dispute / governance console
  - 完整 admin governance backend
  - 项目交易运营台
- 当前最小切分应理解为：
  - `我的项目`
    - 继续承接项目资产与项目推进
  - 公域交易主线
    - 继续承接公域交易对象与交易主流程
  - `V2.1`
    - 只承接 status / rule / handoff layer
    - 以私域 status visibility 和规则解释为上限

### E.4 Candidate My-building Entry Direction

- 若 `V2.1` 未来进入 `我的楼`，当前最多只允许作为：
  - 一个 `constrained status / rule handoff family`
- 当前最稳妥的方向判断是：
  - 倾向 `我的信用与约束`
- 当前不建议直接冻结为：
  - `我的保证金`
- 原因写死如下：
  - `我的保证金` 只覆盖 `V2.1` 的一部分
  - `我的保证金` 容易过早滑向 payment / funds semantics
  - `我的信用与约束` 更符合当前 `rule-and-status boundary package` 的最小 meaning
- 但上述方向当前仍然只是：
  - bounded private entry direction
  - not runtime final IA truth

## F. Excluded Scope At Boundary-Judgment Level

- 当前必须明确排除：
  - 具体保证金金额冻结
  - 具体费率、扣罚、赔付金额冻结
  - 实际支付执行
  - 账单 / 发票 / 结算
  - 风控评分引擎
  - dispute 流程细则
  - admin console 细则
  - frontend IA 定稿
  - runtime implementation unlock

## G. Truth-owner Rule

- 当前再次写死：
  - 入口 owner 可以归 `我的楼 / profile`
  - truth owner 不自动归 `profile`
  - 若未来存在 `信用 / 保证金 / 交易保障` truth，
    - 仍应由 `Server` 侧相应业务 family 持有
  - 不得让 `BFF` 持有交易保障真相
  - 不得让 `BFF` 持有保证金真相
  - 不得让 `BFF` 持有信用约束真相

## H. My Building Drift Guard

- 当前必须写死：
  - `我的楼` 不得因为 `V2.1` 漂成第二 dashboard
  - `我的楼` 不得因为 `V2.1` 漂成交易运营台
  - `我的楼` 不得因为 `V2.1` 漂成治理后台
  - `我的项目`
  - `我的论坛`
  - `设置`
    的现有家族不得被抹掉或降级
- `V2.1` 若未来进入 `我的楼`，也只允许作为：
  - 一个 bounded status / rule handoff
  - not a second operations console

## I. Dependency Rule

- 当前明确如下：
  - 若 `V2.1` 触及真实资金冻结、扣罚、赔付、退款、代收、清算，
    - 则这些内容当前只能被标记为：
      - `requires V2.2 payment/billing package dependency`
- 当前不得在 `V2.1 boundary judgment` 中偷写成：
  - 当前可独立实现
  - 当前已 payment-ready
  - 当前已 billing-ready
  - 当前已 settlement-ready
- 现有 `blacklist / whitelist / permanent-ban` 文书当前也只允许被引用为：
  - 治理约束与限制状态的 baseline material
  - not current `V2.1` package truth itself

## J. Formal Conclusion

- 当前正式结论如下：
  - `V2.1 信用 / 保证金 / 交易保障` 作为《我的楼》当前合法下一包成立，但当前只允许进入 `minimum package boundary freeze`
- 当前结论不代表：
  - 已 rules freeze
  - 已 contracts ready
  - 已 implementation ready
  - 已 payment ready
  - 已 launch ready
