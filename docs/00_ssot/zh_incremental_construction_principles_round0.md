# 《增量施工原则》

## 原则 1：先盘点再施工
- 任何实现动作都必须建立在当前资产盘点与复核通过之上。
- 先写 docs-only 决策，再推进角色施工。

## 原则 2：增量、可追溯、可关闭
- 只改本轮目标范围；
- 所有改动以已冻结真相为来源；
- 每一项必须有“起点-目标-验收-阻断”闭环。

## 原则 3：禁止重做
- 本地已有有效资产不允许重写同名模块；
- 不得搭建平行版本（parallel stack）；
- 不得把历史已完成工作当作“无效重新开始”。

## 原则 4：边界优先
- 先固定边界，再推进接口，再推进消费；
- `/api/app/*` 与 `/server/admin/*` 为约束边界；
- BFF 不持有业务真相，Server 持有业务真相。

## 原则 5：阶段化执行
- Round 0 = 盘点；
- Round 1 = 实施前复核后的增量实现；
- 无独立复核，不进入 release-prep 与 release。

## 原则 6：旧资产复用
- 优先复用既有页面/路由/接口/契约；
- 复用不足的模块必须先形成缺口清单，不得临时补丁化重建。
