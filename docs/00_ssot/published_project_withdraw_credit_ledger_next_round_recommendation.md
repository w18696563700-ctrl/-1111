# 已发布撤回信用分账本建议

状态：下一轮建议，未实现

## 1. 结论

已发布项目撤回到预发布，建议进入信用账本治理，但不要直接做成不可恢复的硬扣分。

推荐路径：
- 首次撤回：记录信用风险事件，可轻扣或只预警。
- 多次撤回 / 已有竞标方参与后撤回：扣信用分。
- 有正当理由并完成补正后：允许部分恢复。
- 恢复必须走账本冲正，不得直接覆盖当前信用分。

## 2. 当前最小闭环

下一轮最小实现建议：

- Server 新增信用事件：`published_project_withdrawn_to_prepublish`。
- 事件记录：
  - `projectId`
  - `publisherOrganizationId`
  - `actorId`
  - `fromState=published`
  - `toState=submitted`
  - `withdrawReason`
  - `bidderImpactSnapshot`
  - `creditDelta`
  - `reversibleUntil`
- 撤回时 Server 写入信用账本。
- BFF 只透传撤回风险摘要和结果。
- Flutter 只展示风险、原因选择和结果，不计算信用分。

## 3. 扣分建议

建议按影响程度分层：

| 场景 | 建议处理 |
| --- | --- |
| 刚发布且无竞标方查看 / 无竞标记录 | 仅记录风险事件，0 分或 -1 分 |
| 已有竞标方查看但无正式报价 | -2 分 |
| 已有正式报价 / 报名 | -3 到 -5 分 |
| 频繁撤回 | 叠加扣分并进入风控提示 |
| 平台原因、资料误审、不可抗力 | 记录事件但不扣分，或可全额恢复 |

## 4. 恢复建议

信用恢复不能靠“把分加回去”这种覆盖式写法，必须走账本。

推荐恢复方式：
- 补正后重新发布并稳定保留一段时间：自动恢复部分分值。
- 撤回原因被平台审核通过：管理员冲正。
- 竞标方未受影响：可自动冲正。
- 用户连续完成若干次正常履约：通过正向信用事件恢复。

恢复事件建议：
- `published_project_withdraw_credit_reversed`
- `published_project_republished_stable`
- `platform_manual_credit_adjustment`
- `successful_fulfillment_credit_recovery`

## 5. 不建议做法

不建议：
- 前端直接扣分。
- BFF 计算信用分。
- 撤回按钮只弹提示但 Server 不记账。
- 用一个 `creditScore` 字段直接覆盖。
- 没有恢复机制就重扣分。
- 把所有撤回一律重罚，容易打击早期留存。

## 6. 三层拆分

当前最小闭环：
- Flutter 风险提示。
- Server 下一轮写信用事件。
- 不做复杂申诉中心。

需要保留但暂不开通：
- 申诉。
- 人工审核。
- 自动恢复规则。
- 信用分明细页。

后续扩展位：
- 信用账本明细。
- 信用恢复任务。
- 风控分层。
- 竞标方影响评估。
- 运营后台信用调整。

## 7. 四类判断

- 最稳：Server 信用账本事件 + 可冲正恢复。
- 最省成本：本轮只做 Flutter 风险提示，下一轮先记录事件不立即重罚。
- 最适合当前阶段：轻扣或预警，保护内测留存，同时保留信用治理真相。
- 风险最大：前端假扣分或一次性做完整信用处罚 / 申诉 / 风控大系统。
