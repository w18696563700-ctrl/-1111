# 功能状态总表 V1 模板

## A. 目的

本模板用于把每个功能统一登记到一张总表里，防止以下混淆：

- 入口已经挂出，但真实业务尚未实现
- 有部分真实能力，但尚未形成完整闭环
- 文书已经冻结，但代码尚未开工
- 功能当前阻断、延期或取消，却在页面上被误读为“可用”

本模板的原则只有一句话：

- 功能可以先上墙，但真相不能造假

## B. 两条状态轴

每个功能必须同时登记两条状态轴，不允许混成一列：

1. `功能状态`
2. `文书状态`

### B1. 功能状态允许值

| 功能状态 | 含义 | 允许对外口径 |
| --- | --- | --- |
| 已完成 | 已有真实能力、已形成当前范围内闭环、已有验证证据 | 可说“已完成” |
| 受控可用 | 有部分真实能力，能稳定工作，但不是完整闭环 | 可说“部分可用 / 受控可用” |
| 占位已挂出 | 页面、入口、结构已存在，但真实业务尚未落地 | 可说“暂未开放 / 占位中” |
| 阻断中 | 本来计划推进，但被当前 blocker 卡住 | 可说“处理中 / 阻断中” |
| 明确延期 | 当前轮次不做，但已登记、未丢失 | 可说“暂缓 / 后续开启” |
| 已废弃 | 已明确不再做或被新方案替代 | 可说“已下线 / 已废弃” |

### B2. 文书状态允许值

| 文书状态 | 含义 |
| --- | --- |
| 未冻结 | 仍在讨论，边界未定，不允许直接实现 |
| 已冻结待实现 | 规则、边界、contracts 已写死，后续可按冻结内容开工 |
| 已解锁实施 | 已完成实现解锁，可进入代码与联调 |
| 已归档 | 当前文书链已完成或被后续文书替代 |

## C. 强制判定规则

### C1. 何时算 `已完成`

必须同时满足：

- 当前轮次功能边界已明确
- 真实 Server truth 已落地
- App 或 Admin 已能稳定消费
- 当前范围内关键路径已验证
- 有结果证据可回溯

只要缺任一项，就不得标 `已完成`。

### C2. 何时算 `受控可用`

满足以下任意一种可标：

- 已有部分真实能力，但后续动作仍需 handoff
- 当前只实现 read-only boundary，不含 command/execution
- 当前只开放最小闭环，不含扩展链路

示例：

- `支付与账单状态` 当前只实现 `status / explanation / handoff / dependency`

### C3. 何时算 `占位已挂出`

满足以下条件：

- 入口已在产品可见
- 页面结构已存在
- 但没有真实业务真值或真实执行链路

不得把 `占位已挂出` 写成 `受控可用`。

### C4. 何时算 `阻断中`

满足以下条件：

- 该功能本轮本应推进
- 当前存在明确 blocker
- blocker 未解除前不能继续闭环

示例：

- 缺 migration
- 缺 reviewer 真会话
- 缺 provider 凭证
- 云端 active artifact 未对齐

### C5. 何时算 `明确延期`

满足以下条件：

- 当前轮次不做
- 已正式登记，不会丢
- 有明确“后续开启条件”

### C6. 何时算 `已废弃`

满足以下条件：

- 功能已被新方案替代，或被明确取消
- 页面与总表后续不应再继续推进该项

## D. 总表字段

总表至少包含以下字段：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 功能模块 | 是 | 一级模块，如 `支付与账单` |
| 子功能 | 是 | 二级能力，如 `支付状态页` |
| 功能状态 | 是 | 只能用本模板定义的允许值 |
| 文书状态 | 是 | 只能用本模板定义的允许值 |
| 当前已完成 | 是 | 只写真实已完成内容 |
| 当前未完成 | 是 | 只写真实未完成内容 |
| 当前阻断项 | 否 | 若无 blocker 可写 `无` |
| 当前依赖项 | 否 | 外部能力、上游模块、会话、provider 等 |
| 入口是否已挂出 | 是 | `是 / 否` |
| 真相 owner | 是 | `Server / Admin / BFF / Flutter` 中的真实 owner |
| 证据链接 | 是 | 文书、接口、测试、云端回执、日志之一或多项 |
| 最近验证时间 | 是 | 使用绝对时间 |
| 下次开启条件 | 是 | 后续进入下一步的唯一条件 |

## E. 总表模板

复制以下表头即可开始登记：

| 功能模块 | 子功能 | 功能状态 | 文书状态 | 当前已完成 | 当前未完成 | 当前阻断项 | 当前依赖项 | 入口是否已挂出 | 真相 owner | 证据链接 | 最近验证时间 | 下次开启条件 |

## G. 页面展示规则

产品页可以挂入口，但不得伪装为已完成。

面向用户的状态页建议只显示以下 6 行：

- 功能名称
- 当前状态
- 当前已完成内容
- 当前未完成内容
- 当前依赖项
- 后续开启条件

内部总表可继续保留完整字段，不要求全部暴露给用户。

## H. 文案映射建议

内部 `功能状态` 不等于用户文案。

建议对用户只显示以下简化口径：

| 内部状态 | 对用户文案建议 |
| --- | --- |
| 已完成 | 已可用 |
| 受控可用 | 部分可用 |
| 占位已挂出 | 暂未开放 |
| 阻断中 | 处理中 |
| 明确延期 | 后续开启 |
| 已废弃 | 已下线 |

## I. 使用要求

- 入口挂出不等于功能可用。
- 有页面不等于有真值。
- 有文书不等于可上线。
- `功能状态` 与 `文书状态` 不得合并。
- 每次联调回执后，必须回写：
  - `功能状态`
  - `当前已完成`
  - `当前未完成`
  - `当前阻断项`
  - `证据链接`
  - `最近验证时间`

## J. 当前推荐执行顺序

后续每个大功能按以下顺序推进：

1. 先确定文书状态
2. 再登记总表
3. 再决定是否挂入口
4. 再进入实现与联调
5. 最后用真实证据更新状态

## K. 一句话标准

- 功能可以先上墙，但真相不能造假。
