# 《enterprise display case library and change corridor ruling》

## 1. 当前问题

当前 `企业展示工作台` 的案例区、提交门槛和已发布后修改语义，存在四个已经足够明确的问题：

### 1.1 案例区只有“新增案例”，没有“保存案例”语义

当前前端的案例编辑区在用户侧表现为一个持续编辑表单，但正式动作只有：

- `新增案例`

这会让用户自然误判为：

- 当前区只是临时填写区
- 图片和文本会不会自动丢
- 我什么时候算真正“存进去了”

### 1.2 后端内部存在 `draft`，但前台没有正式“草稿箱”承接

当前 `Server` 创建案例后会直接落为：

- `caseStatus = draft`

同时 `application` 也存在：

- `applicationStatus = draft`

但当前用户侧没有一套正式、稳定、可理解的“草稿箱”心智与入口承接。

因此：

- `draft` 可以继续作为 `Server` 内部状态真相存在
- 但不应要求用户在当前 V1 工作台中理解“草稿箱”作为一等对象

### 1.3 提交按钮灰掉时，案例门槛与用户操作之间缺少清晰桥梁

当前工作台提交门槛确实要求：

- 至少有 `1` 个案例

但用户侧看到的是：

- 我已经填了案例
- 也上传了图片
- 但为什么还不能提交

根因不是用户理解差，而是：

- 当前案例编辑器缺少正式“保存到案例库”的动作语义
- 当前页把 `Server` 内部状态与用户编辑行为之间的桥接做得不够清楚

### 1.4 “案例归属”与“修改通道”语义未被冻结

当前如果不明确裁决，后续极易发生两类错误：

1. 把案例库做成 `用户资产`
2. 把已发布展示的修改直接混回当前工作台即时覆盖公域

这两条都不允许成立。

## 2. 对象裁决

### 2.1 企业展示是组织资产，不是个人资产

正式裁决：

- `企业展示档` 属于组织
- `企业展示档` 不是个人私有资产
- 人员变动后，展示资产必须继续跟随组织存在

因此：

- 当前工作台中的基础资料、板块画像、案例、联系人，都不得按 `userId` 建模为个人私有真相

### 2.2 案例库属于企业展示档，不属于单个用户

正式裁决：

- 每个 `board 级 enterprise listing` 拥有一套独立案例库

即：

- `优秀公司` 档有自己的案例库
- `优秀工厂` 档有自己的案例库
- `优秀供应商` 档有自己的案例库

案例库归属正式为：

- `organization-scoped`
- `listing-owned`
- `board-specific`

不允许被解释为：

- 每个用户一套案例库
- 用户离职后案例跟人走
- 多人协作时彼此看不到同一套案例资产

### 2.3 当前组织内有权限的成员共同维护案例库

正式裁决：

- 案例库应由当前组织内有权限的成员共同维护
- 它不是“我个人的案例收藏夹”
- 它也不是“只有创建人能继续改”的个人草稿箱

## 3. 用户侧案例语义裁决

### 3.1 前台不把“草稿箱”作为主心智暴露

正式裁决：

- `draft` 可以继续存在于 `Server` truth
- 但当前 `Flutter App` 不应把“草稿箱”作为用户主语义对象暴露

当前用户可见语义应收口为：

1. `未保存`
2. `已保存到案例库`
3. `已进入审核 / 已进入公域`

不要求用户理解：

- `draft`
- `pending_review`
- `edit draft`
- `application draft`

这些术语可以继续作为后台和 contract 的内部状态层存在，但不应成为当前工作台的主要用户语言。

### 3.2 当前案例编辑区应改名为“案例编辑器”，并具有正式保存动作

正式裁决：

- 当前工作台里的“新增案例”区，不应只表现为一次性创建按钮
- 它应被解释为：
  - `案例编辑器`

必须具备的正式动作：

- `保存案例`

该动作的用户语义是：

- 把当前填写的案例标题、类型、城市、日期、摘要、图片、重点标记
  正式保存进当前展示档的案例库

### 3.3 提交门槛只认“已保存进案例库”的案例

正式裁决：

- 提交入驻申请时，`hasCase` 只能由 `Server` 持久化结果决定
- 不得由前端当前编辑框是否有内容来决定

也就是说：

- 用户只是在输入框里写了内容，不算完成案例门槛
- 用户点击 `保存案例` 后，案例写入当前案例库，才算满足案例门槛

## 4. 当前工作台信息架构裁决

### 4.1 案例区应拆成两个相邻区块

当前工作台的案例部分正式拆为：

1. `案例编辑器`
2. `案例库`

#### 案例编辑器

只承担：

- 当前要新建或继续编辑的单个案例内容输入

必须提供：

- `保存案例`

#### 案例库

只承担：

- 当前展示档下已经保存的案例列表回读

必须明确告诉用户：

- 提交门槛认的是这里
- 不是上面的临时输入框

### 4.2 “已有案例”应改成更准确的用户语言

正式建议语言：

- `案例库`
- 或 `已保存案例`

不建议继续只写：

- `已有案例`

因为这会弱化“这是正式资产列表”的语义。

### 4.3 案例保存成功后的用户提示必须改义

不应只提示：

- `案例已新增`

应改成更接近真实心智的语言，例如：

- `案例已保存到当前展示档`
- `当前案例已进入案例库`

## 5. 已发布展示的修改通道裁决

### 5.1 企业展示工作台必须存在正式修改通道

正式裁决：

- 工作台不只是首次入驻页
- 工作台必须继续承担“已存在展示档的维护与修改入口”

### 5.2 未发布态与已发布态必须区分

正式裁决：

#### 未发布态

- 可以继续直接编辑
- 保存后更新私域当前展示档
- 不涉及公域变更治理

#### 已发布态

- 不应把当前工作台简单解释成“直接改线上公域”
- 应进入正式的 `变更提交通道`

### 5.3 已发布态的正确语义不是“直接改公域”，而是“提交变更”

正式裁决：

- 已发布展示的修改，语义上属于 `展示变更`
- 不是“即时覆盖公域”

因此已发布态的正确链路应冻结为：

1. 在工作台继续编辑
2. 形成当前展示档的变更内容
3. 提交变更
4. 进入平台治理 / 审核 / 发布
5. 再更新公域展示

### 5.4 当前轮不冻结频次规则

本轮明确冻结：

- 需要 `修改通道`
- 需要 `已发布态 -> 变更提交` 的治理边界

本轮明确暂不冻结：

- 一个自然月几次
- 会员每天几次
- 是否存在付费加速或附加额度

频次规则延后单独裁决。

## 6. 当前 contract / code 缺口

当前仓库已经具备：

- `create case`
- `delete case`
- `workbench readback`

但当前仍缺：

1. 正式的 `update case`
2. 正式的 `read single case detail / resume editing`
3. 已发布态的正式 `change request` contract

因此本轮正式结论是：

### 6.1 可以立即推进的

- 前端用户语言收口
- 案例编辑器与案例库分层
- `保存案例` 动作语义收口
- 当前案例门槛解释收口

### 6.2 不能假装已经存在的

- 案例完整编辑续写链
- 发布后变更申请链
- 变更审核频次规则

## 7. 实施顺序裁决

按正式顺序应分两段推进：

### 7.1 第一段：案例区用户语义闭环

目标：

1. 案例编辑器 + 保存案例
2. 案例库正式命名
3. 不再让用户承担“草稿箱”概念
4. 提交区明确说明“至少 1 个已保存案例”

### 7.2 第二段：已发布展示修改通道闭环

目标：

1. 冻结 change corridor truth
2. 补 contract
3. 补 backend truth
4. 再补前端消费

## 8. 本轮总控结论

正式结论如下：

- `案例库` 必须跟 `企业展示档 / 公司` 走，不得跟 `用户` 走
- `草稿` 可以继续留在 `Server` 内部，但不得作为当前用户侧主语言
- 工作台案例区必须有 `保存案例` 动作
- 提交门槛只认“已保存到案例库”的案例
- 企业展示工作台必须具备正式修改通道
- 已发布态的修改必须走 `变更提交`，但频次规则本轮暂不冻结
