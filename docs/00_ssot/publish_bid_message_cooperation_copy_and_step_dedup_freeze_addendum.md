# 《发布-竞标-沟通-合作确认链路词表与步骤去重收口冻结单 V1》

## 1. 总裁决

本文件冻结当前普通展览项目链路的用户主体验词表与步骤去重规则。

当前最稳的方案：

1. 先冻结词表、步骤和禁用入口。
2. 再只做 Flutter 展示层最小收口。
3. 不先补局部 UI，不让历史收费、订单、消息楼口径继续混用。

当前最低成本的方案：

1. 不改 BFF、Server、contracts、OpenAPI、数据库、云端部署。
2. 不重做竞标流程、不新增页面族、不新增业务命令。
3. 只改用户可见文案、步骤标题、入口标签和测试断言。

当前最适合当前阶段的方案：

1. 先解决 `项目详情 -> 竞标提交` 的重复强核对。
2. 先解决旧 `P0-Pay / 发单诚意金 / 询价报价单` 口径残留。
3. 先解决消息楼被误读为支付台、订单台、裁决台的入口问题。

当前风险最大的方案：

1. 继续局部补 UI，最后再统一梳理。
2. 让 `申请参与`、`继续竞标`、`提交竞标`、`合作确认`、`订单状态` 继续混用。
3. 让普通用户误以为消息楼可以直接支付、定标、裁决或完成成交确认。

## 2. 当前最小闭环

当前普通展览项目最小闭环固定为：

`创建项目 -> 预发布补资料 -> 200 元项目真实性诚意金 -> 正式发布 -> 申请参与竞标 -> 审核通过 -> 4000 元竞标服务费预授权额度 -> 立即参与竞标 -> 提交竞标 -> 项目沟通 -> 发起合作确认 -> BidAward -> Order / Contract seed -> 后续成交确认`

解释固定如下：

1. `申请参与竞标` 只表示竞标准入申请，不等于提交竞标。
2. `审核通过` 只表示准入通过，不等于可以跳过 `4000 元竞标服务费预授权额度`。
3. `立即参与竞标` 是从项目详情进入竞标提交页的动作，不等于首次准入申请，也不等于最终提交。
4. `提交竞标` 只能表示执行既有 `POST /api/app/bid/submit` 主命令。
5. `项目沟通` 只承接沟通、状态通知、只读摘要和软链接。
6. `发起合作确认` 是发布方选择候选竞标方并进入后续承接的动作。
7. `BidAward -> Order / Contract seed` 只是后续承接锚点生成，不等于成交成立。
8. 正式成交成立必须等待后续单独冻结的成交确认链路，不由本轮 Flutter 文案宣称完成。

## 3. 需要保留但暂不开通

以下能力必须保留为平台能力或后续承接位，但不得作为本轮用户主体验入口：

1. 完整订单工作台。
2. 完整合同确认工作台。
3. 里程碑、验收、评价、售后。
4. 消息楼内直接支付。
5. 消息楼内直接定标。
6. 消息楼内直接成交确认。
7. 候选方比较台、评分台、报价对比矩阵。
8. BidAward 后的完整成交确认和扣费释放 UI。
9. `converted_to_order` 后报价依据资料可见性的 Server/BFF 解锁修复。
10. 底层历史 `p0_pay` 命名、历史文书、兼容测试和内部实现痕迹。

## 4. 后续扩展位

后续扩展位固定为：

1. 成交确认页：承接双方确认、成交金额、附件和最终平台服务费结果。
2. 消息楼确认卡：只承接软链接、状态摘要和来源消息，不直接写业务状态。
3. 候选比较面板：在候选资料、合同承接和比较规则冻结后再开。
4. 资金状态面板：只读聚合 `200 元项目真实性诚意金`、`4000 元竞标服务费预授权额度`、扣费、释放和退款状态。
5. 后续承接状态入口：只在已有真实 `orderId / contractId / dealConfirmationId` 时作为软链接或只读入口展示。

## 5. 三组正式词表

### 5.1 准入词表

用于 `项目详情 -> 申请参与竞标 -> 审核` 段。

| 场景 | 正式主文案 | 不得表达成 |
| --- | --- | --- |
| 首次申请 | `申请参与竞标` | `立即竞标`、`提交竞标`、`报价` |
| 申请状态 | `参与竞标申请` | `项目名称查看申请` 作为新入口 |
| 审核通过 | `准入已通过` / `参与竞标申请已通过` | `已提交竞标`、`已中标` |
| 审核后下一步 | `立即参与竞标` | `继续竞标` 作为模糊 CTA |
| 审核中 | `参与申请审核中` | `竞标中` |

### 5.2 提交词表

用于 `立即参与竞标 -> 查看材料 -> 填写报价 -> 上传方案 -> 提交竞标` 段。

| 场景 | 正式主文案 | 不得表达成 |
| --- | --- | --- |
| 项目承接摘要 | `已承接项目` | `第一步 核对项目` 作为重复强步骤 |
| 可选复核 | `复核项目信息` | `重新展开核对` |
| 从详情进入提交页 | `立即参与竞标` | `继续提交竞标`、`继续竞标` |
| 最终动作 | `提交竞标` | `确认服务费规则并继续提交` |
| 材料读取 | `查看报价依据资料` | `项目附件` 作为主步骤 |
| 4000 说明 | `竞标服务费预授权额度` | `平台服务费预授权`、`预计服务费` |
| 技术说明 | `以平台记录为准` | `Flutter 不本地计算服务费` |

### 5.3 合作确认词表

用于 `项目沟通 -> 发起合作确认 -> 后续承接` 段。

| 场景 | 正式主文案 | 不得表达成 |
| --- | --- | --- |
| 发布方选择候选方 | `发起合作确认` | `轻量选择`、`立即成交` |
| 业务命令按钮 | `确认合作并生成订单` | `定标完成`、`成交成立` |
| 命令说明 | `生成后续订单/合同承接锚点，不等于成交成立` | `合同已确认`、`已完成成交` |
| 消息楼入口 | `项目沟通` / `竞标沟通` / `结果通知` | `支付台`、`订单台`、`裁决台` |
| 已有真实订单锚点 | `后续承接状态` / `后续承接入口` | `订单台`、`订单工作台` |

## 6. 禁用词与替换矩阵

以下词不得作为普通用户主体验解释主线：

| 禁用词 | 本轮替换口径 | 允许保留位置 |
| --- | --- | --- |
| `P0-Pay` | `平台收费规则` 或具体对象名 | 历史文书、底层类名、兼容测试名 |
| `发单诚意金` | `项目真实性诚意金` | 历史文书、兼容解析、旧数据迁移说明 |
| `询价报价单` | `普通展览项目` 或 `报价依据资料` | 历史 P0-Pay 文书、兼容代码 |
| `平台服务费预授权` | `竞标服务费预授权额度` | 历史兼容测试、非用户主体验内部注释 |
| `预计服务费` | `成交后按平台规则处理` | 后续成交确认冻结后可由上游返回正式金额 |
| `支付台` | 不提供该入口 | 不允许作为当前 App 主体验 |
| `订单台` | `后续承接状态` / `后续承接入口` | 后续完整订单工作台单独冻结后再开 |
| `裁决台` | 不提供该入口 | 不允许作为消息楼主体验 |
| `routeTarget` | `后续入口` 或不展示 | parser、registry、测试排查上下文 |
| `handoff` | `承接入口` 或不展示 | 技术文书、测试排查上下文 |
| `Flutter 不本地计算服务费` | `以平台记录为准` | 代码注释或开发者排查，不进普通用户主体验 |

## 7. 步骤去重裁决

### 7.1 项目详情

项目详情已经展示项目概要、地点安排、预算、状态和参与入口。

裁决：

1. 项目详情承担首次项目信息浏览和进入竞标提交页的入口职责。
2. 项目详情按钮统一收口为 `立即参与竞标`。
3. 项目详情不得把 `立即参与竞标` 写成首次准入申请或最终提交成功。

### 7.2 竞标提交页

竞标提交页不得再次以强制 `第一步 核对项目` 重复项目详情的完整核对。

裁决：

1. 竞标提交页首屏主标题改为 `已承接项目`。
2. 首屏只展示最小项目摘要和状态。
3. 完整项目信息作为可选 `复核项目信息`。
4. 后续步骤按 `查看报价依据资料 -> 填写报价与预授权确认 -> 上传方案 -> 提交竞标` 展开。
5. 若缺少项目上下文，仍允许展示阻断信息，但不得伪装为可提交。

### 7.3 项目沟通

消息楼不得承接支付、订单、裁决、成交确认业务真相。

裁决：

1. 消息楼列表只展示 `项目沟通`。
2. 项目沟通页可展示 `竞标沟通`、`参与竞标申请`、`项目相册`、`后续承接状态` 等软入口。
3. `订单状态` 在消息楼主体验中降级为 `后续承接状态`。
4. 如果没有真实 `orderId`，不得展示可点击订单入口。
5. 如果有真实 `orderId`，入口也只能是后续承接软链接，不等于消息楼成为订单台。

### 7.4 合作确认

合作确认只允许在发布方候选竞标方场景出现。

裁决：

1. 候选卡主动作可保留业务命令 `确认合作并生成订单`。
2. 标题和说明必须明确这是 `发起合作确认`。
3. 必须提示 `生成后续订单/合同承接锚点，不等于成交成立`。
4. 不新增候选比较台、评分台、合同确认台、成交确认台。

第 1 天文书验收标准：

1. 当前最小闭环已写清。
2. 需要保留但暂不开通已写清。
3. 后续扩展位已写清。

1. 普通用户主体验不出现 `P0-Pay`、`发单诚意金`、`询价报价单`。

5. 合作确认不宣称成交成立。
6. 最终提交仍只走既有 `POST /api/app/bid/submit`。

第 3 天验证验收标准：

1. 定向 Flutter tests 通过，或失败项明确登记为非本轮旧债。
2. `git diff --check` 覆盖本轮改动文件并通过。
3. 无 BFF、Server、contracts、OpenAPI、云端改动。
4. 最终回执明确本轮不代表云端 runtime 完成。

## 10. No-Go 条件

出现以下任一情况，不得进入下一天：

1. 用户未确认本文件口径。
2. 合作确认、订单/合同锚点、消息楼边界仍存在歧义。
3. 需要新增接口、字段、状态、枚举或命令。
4. 需要修改 BFF、Server、contracts、OpenAPI、数据库或云端 runtime。
5. Flutter 展示层改动需要伪造业务真相。
6. 消息楼被设计成支付台、订单台、裁决台或成交确认台。
